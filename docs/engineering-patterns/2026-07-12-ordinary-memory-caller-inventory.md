```powershell

```

The post-audit AST normalization yields 44 unique `(path, enclosing symbol, writer)` tuples
and 48 writer occurrences. Creation-only calls are intentionally absent from the mutation
writer set; tuple identity and multiplicity remain guarded without recording line numbers.

Search-only, non-caller matches are classified as follows:

- `ContextEngine._persist_ordinary_memory` at its definition is an implementation symbol,
  not a call.
- The `update_memory_fields()` mention in `ContextEngine.get_memory_dict` is documentation.
- Definitions of `update_memory_fields`, `increment_field`, `batch_update`, and
  `delete_memory` are not calls. Their internal calls and raw SQL remain in the table.
- Direct `patch_ordinary_memory()` calls and constant-name `getattr` aliases are resolved to
  their canonical writer. This keeps both explicit and compatibility field patches visible.

## Classification Rules

- `occurrences` is the positive count of matching calls or SQL statements in that exact
  path/symbol/writer tuple. Adding or removing a repeated occurrence requires inventory review.
- A local alias is classified only when its assignment proves a canonical writer through a
  constant-name `getattr` or direct writer attribute. Dynamic or reassigned aliases are ignored.
- `existing_id=yes` means the operation targets a canonical row already known to exist.
  `possible` includes caller-supplied IDs, deterministic IDs, replay, and len/time-derived IDs.
- Ordinary rows name exactly one intended authority. Mixed functions name the authority for
  the risky existing-ID branch; their semantics and migration task explicitly require the
  create/metadata branches to be split.
- The two `SynthesisStore` rows are proven synthesis-only (`memory_type='synthesis'` plus the
  synthesis control transaction). They stay visible to the raw-SQL guard but are marked as a
  non-ordinary authority instead of being dishonestly routed through an ordinary-memory API.
- Proposal promotion is the reviewed-safe exception: its deterministic ordinary insert occurs
  only on a dedup miss inside the proposal, lineage, version, and outbox transaction. Task 6
  must retain that transaction-local insert.

Status values are `pending`, `reviewed-safe`, and `reviewed-nonordinary`; the registry test
rejects every unresolved `pending` row. The current registry contains 0 `pending`,
42 `reviewed-safe`, and 2 `reviewed-nonordinary` tuples.

## Current Conclusions

- Task 2 migrated ordinary feedback, metadata reset, duplicate reinforcement, allowed pipeline
  patches, and all four daemon tag writes to canonical field-scoped CAS writes.
- Task 5 installed the source coordinator and routed public content/correction/forget,
  RecMem update/forget, skill completion content, audit rollover, duplicate cleanup, memory
  merge, and smart-remember duplicate replacement through checked atomic mutations.
- Task 6 made whole-record ordinary storage creation-only. Identical full canonical bindings
  replay without writes; conflicting caller-supplied, pack, deterministic skill, bootstrap,
  migration, audit, and review IDs raise without replacing the existing row.
- Pack replace now coordinates content plus pack-owned policy fields in the source invalidation
  transaction. Pack merge remains a narrow CAS field patch, availability-changing pack tags
  fail before either mutation path, and missing pack rows use creation.
- Reclassification now commits its classification patch and checked `memory-index/v3` upsert
  in one SQLite transaction; enqueue/material failure rolls back the patch and memory version.
- No writer row remains pending. The only storage-owned startup update is bounded and tested;
  the retained private `_upsert` and generic `delete` forms have no production caller and are
  guarded by the AST inventory against silent introduction.
- Duplicate-cluster cleanup now uses
  `ContextEngine.mutate_ordinary_source(operation="forgotten",
  reason="safety-net:duplicate_cluster")`; neither raw update nor hard delete remains.
- Proposal promotion's insert is already transactionally safe and must not be extracted from
  its proposal, lineage, version, and durable-index transaction.

This inventory is also a release verification input; release-sync ships this
record with the production tests that assert the migration evidence.

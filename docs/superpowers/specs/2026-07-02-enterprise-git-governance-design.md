# Enterprise Git Governance — Plastic Promise Flow

> **Date**: 2026-07-02
> **Status**: Approved — Implementation in progress
> **Paradigm**: 约定工程 (Commitment Engineering) — 内化约定替代外部约束

## Overview

Enterprise-grade Git governance framework purpose-built for hybrid human + AI Agent collaboration. Integrates the Hunter Guild delegation system as the orchestration layer, with Git as the versioning layer and Pull Requests as the interface between them.

### Core Insight

> A Pull Request IS a Hunter Guild delegation — it carries the same lifecycle: claim → implement → review → verify → close. Trust scores become merge gates.

```
Hunter Guild (编排层)
  Daemon scanners → task_enqueue → Hunter claim →
  → Agent implements → task_complete → Claude elder verify
                       │
            PR is the delegation's Git carrier
                       │
Git Workflow (版本层)
  feat/* branch → git push → PR created → CI checks →
  → Code Review → Squash Merge → main
```

## 1. Architecture — Git + Hunter Guild Integration

### 1.1 Key Mappings

| Hunter Guild Concept | Git Concept | Description |
|----------------------|-------------|-------------|
| task_enqueue | Issue creation | Each delegation corresponds to an Issue; implementation artifact corresponds to a PR |
| task_claim | Branch creation | Agent creates worktree/feature branch and begins work |
| task_complete | PR submission | Agent pushes branch and creates PR |
| task_verify | Code Review | Claude (or human) reviews PR; approved = merged |
| Trust score | Merge permissions | S-level (≥0.80) autonomous merge; A-level (≥0.65) needs 1 approve; rest need 2 approves |

### 1.2 Trust Score — Full Lifecycle Changes

| Event | Git Trigger | Trust Delta | Target |
|-------|------------|-------------|--------|
| Scanner detects problem | — | -0.01 ~ -0.03 | Problem introducer (Agent/module) |
| Hunter claims task | Branch created | no change | — |
| Fix complete + CI passes | PR submitted | +0.02 | Fixer |
| Elder rejection | PR Review rejected | -0.03 | Fixer (cumulative with original deduction) |
| Verification approved | PR Merged | +0.02 | Fixer |

### 1.3 Problem Attribution Rules

```
Scanner detects issue:
  → Can pinpoint to specific line → deduct from git blame author of that line
  → Can pinpoint to file but not line → deduct from git blame author of most recent commit on that file
  → Cross-file / cannot precisely locate → average deduction across related module recent authors
  → Cannot attribute at all → record deduction on module (not yet assigned), Claude reviews periodically
```

### 1.4 Auto Branch Naming Hook

```
Agent executes task_claim(task_id, agent_name)
  → Daemon auto-generates branch name:
    <type>/<task_id>-<slug>
    e.g. fix/T42-memory-leak-pipeline
    e.g. feat/T17-add-health-check-endpoint
  → <type> mapping:
    fix_memory / fix_* / fix_ci → fix/
    build_* / feat_*           → feat/
    refactor_*                 → refactor/
    docs_*                     → docs/
    perf_*                     → perf/
  → <slug> auto-truncated from title first 3-4 keywords, kebab-case
  → Agent directly git checkout -b <generated branch name>, zero manual naming
```

task_id embedded in branch name → bidirectional traceability from any git log/blame/PR back to the delegation. Audit chain closed.

## 2. Branch Strategy & Naming

### 2.1 Branch Structure

```
main                              ← Single long-lived branch, always deployable
  │
  ├── feat/<description>          ← New feature (maps task_type: build_*)
  ├── fix/<description>           ← Bug fix (maps task_type: fix_memory / fix_*)
  ├── refactor/<description>      ← Refactor (maps task_type: refactor_*)
  ├── docs/<description>          ← Documentation (maps task_type: docs_*)
  ├── perf/<description>          ← Performance optimization
  ├── chore/<description>         ← Build/tooling/CI config
  │
  └── worktree/<agent>/<description> ← Agent worktree working branch
                                       e.g. worktree/pi_builder/fix-memory-leak
```

### 2.2 Rules

- `main` always protected, direct push forbidden
- Branch names all lowercase, `-` separated
- Worktree branches prefixed `worktree/<agent>/` for identification and batch cleanup
- Merge via **Squash Merge**, keeping `main` history linear and clean
- Each PR merged as one commit, commit message = PR title

### 2.3 Branch Lifecycle

```
Daemon scanner detects issue
  → task_enqueue(type, title, to_agent)
  → Hunter claim → git checkout -b <auto-generated-branch-name>
  → Development → commit (conventional commits)
  → task_complete + git push + PR created
  → CI check passes + Code Review passes
  → Squash Merge → git branch -d <branch>
  → task_verify(accepted) → loop closed
```

### 2.4 Branch Timeout & Cleanup

```
Branch > 7 days unmerged
  → Daemon notifies Agent: "Branch <branch> has timed out without merge, will auto-delete in 24h"
  → 24h later still unmerged → git branch -D <branch>
  → Associated delegation → task_abandon → trust -0.02
```

## 3. Commit Conventions

### 3.1 Specification

All commits MUST follow Conventional Commits: `<type>(<scope>): <subject>`

- `scope` is optional
- `subject` lowercase, no period at end
- English only

### 3.2 Types

| Type | Usage | Maps to Hunter Guild task_type |
|------|-------|-------------------------------|
| `feat:` | New feature | `build_*` |
| `fix:` | Bug fix | `fix_memory` / `fix_*` |
| `refactor:` | Refactor (no behavior change) | `refactor_*` |
| `docs:` | Documentation changes | `docs_*` |
| `perf:` | Performance optimization | `perf_*` |
| `test:` | Tests | `test_*` |
| `chore:` | Build/CI/tooling | `chore_*` |
| `revert:` | Rollback | — |

## 4. CI/CD Pipeline

### 4.1 Pipeline Structure

```
PR Submitted / Updated
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Stage 1: Fast Validation (P0 — MUST PASS)      │
│  ─────────────────────────────────────────────── │
│  - Compile / type check (Python + Rust)          │
│  - Unit tests (pytest)                           │
│  - Security scan (bandit / cargo-audit)          │
│  - Dependency compliance (license, version)      │
│                                                  │
│  All pass → proceed                              │
│  Any fail → block merge, trigger deduction,      │
│             auto-create fix_ci delegation        │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Stage 2: Quality Gates (P1 — SHOULD PASS)      │
│  ─────────────────────────────────────────────── │
│  - Code style check (ruff format --check)        │
│  - Test coverage (≥80% pass, <80% warn)          │
│  - Cyclomatic complexity (per-function ≤15)      │
│                                                  │
│  All pass → auto-advance                         │
│  Warnings → log to audit, allow continue         │
│             (micro-deduction)                    │
│  Severe deviation → mark as suggested fix,       │
│                      does not block merge        │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Stage 3: Performance Baseline (P2 — INFO)      │
│  ─────────────────────────────────────────────── │
│  - Critical path latency diff (vs main baseline) │
│  - Memory usage delta                            │
│  - LanceDB search performance                    │
│                                                  │
│  Log to audit only, no trust score changes       │
│  Used for long-term trend analysis               │
└─────────────────────────────────────────────────┘
```

### 4.2 Trust Score — CI Linkage

| Stage | Trigger | Trust Delta | Action |
|-------|---------|-------------|--------|
| Problem detected | Daemon scanner finds issue | -0.01 ~ -0.03 | Attribute to responsible party, deduct |
| PR submitted | Agent submits PR | no change | CI auto-triggered |
| P0 failure | Compile/test/security any fail | -0.02 | Block merge, auto-create fix_ci delegation (assigned to recent modifier, 30min window) |
| P1 warning | Style/coverage/complexity deviation | -0.005 | Log to audit, does not block merge |
| P0+P1 pass | All required and suggested checks pass | +0.01 | No action, wait for review |
| Code Review | Reviewer raises change request | no change | Agent fixes then resubmits, CI re-runs |
| Merged | PR merged to main | +0.02 | Trust score updated, loop closed |
| Rejected | Reviewer rejects merge | -0.03 | Trigger fix_review delegation, reassign |

### 4.3 CI Failure Auto-Repair

When P0 stage fails, Daemon auto-generates a fix delegation:

```
task_enqueue(
    task_type="fix_ci",
    title="Fix CI failure: <PR title>",
    to_agent="<agent who last modified the file>",
    priority=2,  # A-level — CI failure is blocking
    payload={
        "pr_id": "<PR number>",
        "failed_stage": "compile | test | security",
        "error_log": "<CI log summary>"
    }
)
```

Agent has 30-minute window to fix. Timeout triggers additional deduction (-0.01) and auto-escalates to Claude.

## 5. Code Review Process

### 5.1 Roles & Permissions

| Hunter Guild Role | Git Role | Trust Score | Review Permission |
|-------------------|----------|-------------|-------------------|
| Claude (Elder) | Maintainer | ≥0.80 | Final approval, merge, may self-approve (but 1 other approve recommended) |
| Pi Reviewer | Code Reviewer | ≥0.65 | Review comment, approve; submitted PRs auto-assigned reviewer |
| Pi Builder | Contributor | ≥0.50 | Submit PR, respond to review comments |
| Pi Fixer | Fix Contributor | ≥0.35 | Can only submit fix-type PRs, requires 2 approves |
| Demoted Agent | — | <0.35 | Cannot submit PRs, cannot review |

### 5.2 Review Flow (Standard Path)

```
Agent submits PR
  │
  ├── CI Stage 1 (P0) MUST pass
  ├── CI Stage 2 (P1) SHOULD pass
  │
  ├── Auto-assign reviewer:
  │     trust ≥ 0.65 → Pi Reviewer auto-claims
  │     trust < 0.65 → Claude manually assigns
  │
  ├── Reviewer reviews:
  │     Comment → Agent fixes → Push update → CI re-runs
  │     Loop until Reviewer approves
  │
  ├── Claude final approve (Elder verification)
  │
  └── Squash Merge → task_verify(accepted) → +0.02
```

### 5.3 Review Requirements by PR Source

| PR Source | Review Requirement |
|-----------|-------------------|
| Pi Builder (≥0.50) | 1 Reviewer approve + Claude approve |
| Pi Fixer (≥0.35) | **2 Reviewer approves** + Claude approve |
| Daemon auto PR (fix_ci) | 1 Reviewer approve + Claude approve (30min SLA) |
| Human developer | 1 Reviewer approve + Claude approve |

### 5.4 Review Comment Classification

| Comment Type | Label | Trust Impact |
|-------------|-------|-------------|
| Minor issue (naming, formatting) | `nit` | No deduction, Agent must fix |
| Design issue (architecture, logic) | `design` | -0.005 each |
| Security/Bug risk | `blocking` | -0.01 each, must fix |
| Positive feedback | `praise` | +0.005 each |

## 6. Security Configuration

### 6.1 Branch Protection Rules (`main`)

| Rule | Setting |
|------|---------|
| Direct push forbidden | All changes must go through PR |
| Force push forbidden | History is immutable |
| CI must pass before PR merge | All P0 must pass |
| Code Review required before merge | At least 1 approve |
| CODEOWNERS review | Key paths require at least 1 owner approve |
| Branch deletion forbidden | `main` cannot be deleted |
| Linear history | No merge commits, Squash Merge only |
| Conversations resolved before merge | All comment threads must be resolved |

### 6.2 CODEOWNERS

```
# Global owner — Claude (human)
*                           @ALdaisuki

# Core engine — requires Claude + 1 Reviewer
plastic_promise/memory/     @ALdaisuki
plastic_promise/loop/       @ALdaisuki
plastic_promise/principles/ @ALdaisuki

# Rust engine — requires Claude approve
rust/                       @ALdaisuki

# Security-sensitive — requires Claude sole approve
plastic_promise/defense/soul_enforcer.py @ALdaisuki
plastic_promise/defense/trust_store.py   @ALdaisuki

# Docs and config — Pi Builder may modify
/docs/          @ALdaisuki @pi-builder
*.md            @ALdaisuki @pi-builder
```

### 6.3 Authentication

- SSH key authentication required for all git operations
- HTTPS with PAT (Personal Access Token) as fallback
- No password-based authentication
